# P2 — Burnless Pilot PTY: host-neutral para Claude e Codex

**Objetivo:** `burnless pilot` (alias compatível: `burnless pty`) spawna o CLI interativo escolhido pelo usuário (`claude` ou `codex`) e mantém a TUI nativa intacta. Burnless cuida de journal, checkpoint, contexto e rollover por fora. O host continua dono de autenticação, modelo, plugins, tools, permissões, sandbox e UX.

**História relevante:** o antigo `burnless chat`/`pty_shell`/maestro-REPL foi cortado em 2026-06-14 (`_design/cut_chat_pty_2026_06_14.md`) por nunca ter sido usado — era uma UI de chat própria. O novo modo é DIFERENTE: não reconstrói UI; é um relay transparente + watcher. O corte foi verificado: `pty_shell.py`, `chat_history.py`, `maestro/`, `maestro_layer.py`, `pipeline_state.py` ausentes; **zero `import pty`/`pexpect` no repo** — a primitiva PTY parte do zero.

## Veredito de arquitetura

**Construir um PTY transparente sobre um `HostAdapter`, com respawn-fresh primeiro.** PTY é somente transporte de bytes/TTY. Memória nunca depende de parsing visual da tela. Cada adapter normaliza hooks/transcript/usage/reset para o protocolo Burnless.

**Não pin model by default.** `pilot.model: null` significa omitir `--model`: Claude/Codex usam a configuração/recomendação ativa do próprio CLI. Override explícito continua disponível. Burnless detecta o binário no `PATH`, não atualiza CLIs automaticamente e não promete que uma versão futura desconhecida é compatível sem capability probe.

**Zero atrito é meta testável, não premissa.** Há um onboarding único para escolher host e confiar hooks. Depois disso, a sessão deve parecer o CLI puro. Se capacidades obrigatórias faltarem, auto-rollover desliga com diagnóstico claro; nunca arrisca uma troca.

Evidência local em 2026-07-02: `codex-cli 0.135.0` e Claude Code `2.1.198` estão instalados. Codex atual oferece TUI, `resume`, `fork`, `/clear`, `/new`, hooks `UserPromptSubmit`/`Stop`/`SessionStart(clear)` e config própria; hooks async ainda são ignorados. Claude oferece `--session-id`, `--resume`, `--fork-session` e `SessionEnd(clear)`. Esses contratos são parecidos, não idênticos.

## Contrato `HostAdapter`

Novo pacote sugerido:

```text
src/burnless/pilot/
  core.py
  pty_relay.py
  host.py
  hosts/claude.py
  hosts/codex.py
  events.py
```

Interface mínima:

```python
class HostAdapter(Protocol):
    name: str
    def detect(self) -> HostInstallation: ...
    def capabilities(self) -> HostCapabilities: ...
    def build_interactive_argv(self, root, model=None, extra_args=()) -> list[str]: ...
    def normalize_hook_event(self, payload) -> PilotEvent: ...
    def locate_session(self, run_id) -> HostSession: ...
    def context_usage(self, session) -> ContextUsage: ...
    def is_turn_idle(self, session) -> bool: ...
    def build_fresh_argv(self, root, model=None, extra_args=()) -> list[str]: ...
```

`PilotEvent` usa `host`, `host_session_id`, `process_instance_id`, `event`, `source`, `cwd`, `transcript_ref`, `user_text`, `assistant_text`, `usage`, `ts`. O core P1 recebe somente esse envelope.

### Configuração

```yaml
pilot:
  host: auto                 # auto|claude|codex
  model: null                # null = default/recommended do CLI
  extra_args: []
  rollover_mode: respawn     # respawn|native-clear (depois)
  rollover_at_pct: 0.65
  delta_budget_tokens: 2000
  hud: title                 # title|off; nunca desenhar sobre a TUI
```

Resolução: `--host` > `BURNLESS_PILOT_HOST` > `pilot.host` > escolha persistida. Se ambos estiverem instalados e não houver preferência, `auto` mostra um seletor uma vez; não escolhe silenciosamente por tier/worker. `--model` explícito sobrepõe config. `extra_args` são repassados sem shell interpolation.

### Matriz inicial

| Capacidade | Claude adapter | Codex adapter |
|---|---|---|
| Spawn nativo | `claude` + SID conhecido | `codex -C <root>`; aprende thread id |
| Modelo default | omite `--model` | omite `--model` |
| Turn boundary | hooks/transcript | hooks/rollout, nunca tela |
| Reset inicial | respawn + novo SID | respawn fresh |
| Reset v2 | `/clear` após idle provado | `/clear` ou `/new` após idle provado |
| Lineage | SessionEnd/Start + run id | `BURNLESS_PILOT_RUN_ID` + hooks/sidecar |
| Usage | assistant usage no transcript | hook/rollout/status normalizado |
| Hook async | não necessário | não usar: não suportado atualmente |

## PTY-0 · Pré-requisitos (bloqueantes)

1. **P1 Fases 0A, 0B e 0C completas** — journal síncrono, checkpoint/watermark e handoff/restore aprovados no teste de rewriter lento + `/clear` imediato.
2. **RM-10 aplicado** — fonte do seed = `.burnless/epochs/_rolling/seed.md` → `pending_seed.md` (o `session-*.rollover.md` está morto, sem writer).
3. **Adapter contract + fake host antes do CLI real.** Criar `tests/fixtures/fake_tui_host.py`, capaz de emitir ANSI, resize, turn events, usage e session reset determinísticos.
4. **Config separada de workers.** `pilot.host` não deriva de `agents.silver.provider`; maestro interativo e workers são eixos independentes.
5. **Capability probe.** `burnless pilot doctor` mostra path/version, hooks, trust, transcript/rollout access, reset strategy e suporte a usage para cada host.
6. **Warm-seed separado.** Nunca fork de pool com persona de worker. Default do pilot é fresh; otimização warm só depois de teste de identidade/prompt-cache por adapter.

## PTY-1 · M0 — Relay PTY host-neutral

Implementar `burnless pilot [--host claude|codex]`:
- `pty.openpty()` + relay raw, termios, bracketed paste, Unicode, SIGWINCH e process group.
- Spawn por argv do adapter, sem `shell=True` e sem flags de sandbox/permissão impostas pelo Burnless.
- Definir `BURNLESS_PILOT_RUN_ID=<uuid>` no ambiente do filho; hooks usam esse lineage id em ambas as plataformas.
- Ctrl-C vai ao filho; saída/erro restaura terminal mesmo em exception/SIGTERM. Kill do pilot encerra o process group sem corromper estado.
- Nenhum parsing de ANSI para decisões de memória.

**Verify:** testes automatizados contra fake host cobrem byte fidelity, alternate screen, resize, paste, Ctrl-C, cleanup e exit code. Smoke opt-in abre Claude e Codex e confirma que `/help`, seleção de modelo, approvals e plugins continuam nativos.

## PTY-2 · M1 — Eventos normalizados, contexto e HUD não invasivo

- Cada adapter instala/usa hooks finos que escrevem `PilotEvent` no sidecar `.burnless/pilot/runs/<run_id>/events.jsonl` e chamam o journal síncrono P1.
- Claude lê transcript Claude; Codex lê seus hook events/rollout. Paths e schemas ficam dentro do adapter.
- `context_usage()` retorna valor, limite e confiança (`exact|estimated|unknown`). Auto-rollover por tokens só funciona com confiança aceita; caso contrário usa turns/manual e avisa.
- HUD usa somente title OSC ou integração nativa configurável; nunca uma linha desenhada dentro do alternate screen.
- Limiar configurável: `pty.rollover_at_tokens` (default ~120k) + `pty.rollover_at_pct`.

**Verify:** contract tests alimentam fixtures Claude/Codex e exigem o mesmo `PilotEvent`/`ContextUsage`. Teste garante que payload desconhecido falha para `unknown`, nunca inventa número nem dispara rollover.

## PTY-3 · M2 — Rollover automático (respawn-fresh)

No limiar: (1) adapter prova `idle`; (2) confirma `journal_head`; (3) cria handoff `checkpoint + delta` ligado ao `run_id`; (4) encerra filho gentilmente; (5) spawna `build_fresh_argv`; (6) novo SessionStart reivindica o handoff; (7) HUD/title mostra `↻ host: 143k → checkpoint 3.2k + 1 pending`.
- Nunca rollover no MEIO de um turno (watcher sabe se há turno em andamento pelo jsonl).
- `--no-auto` para rollover só manual (tecla ou comando).
- Se idle, journal ou handoff não puderem ser provados, abortar rollover e manter a sessão viva. Não usar silêncio/mtime como prova.

**Verify:** a mesma suíte parametrizada roda `fake-claude` e `fake-codex`: rollover, rewriter lento, crash em cada boundary, delta exatamente uma vez e terminal recuperado. Depois, smoke real opt-in com limiar baixo nos dois CLIs.

## PTY-4 · M3 — Auditoria de tokens da sessão pilotada

Cada rollover registra em `.burnless/pilot/session_log.jsonl`: `{ts, run_id, host, host_version, old_session, new_session, context_before, context_confidence, checkpoint_chars, pending_count, turns, strategy, duration_ms}`. `burnless pilot --report` imprime a linha do tempo e integra depois com P3.

## PTY-5 · M4 — Variante `/clear` (v2, opcional)

`native-clear` é capability do adapter, não string global. Claude pode usar `/clear`; Codex pode usar `/clear` ou `/new`, conforme capability probe. Só habilitar depois que testes provarem idle, novo session/thread id e SessionStart restore. Default permanece `respawn`.

## Aceitação do modo (fim-a-fim)

1. `burnless pilot --host claude` e `--host codex` abrem TUIs indistinguíveis do host puro nos smoke tests suportados.
2. Com `model: null`, cada host usa sua configuração/modelo atual; override explícito funciona sem contaminar o outro host.
3. Até o limiar, rollover automático continua objetivo/arquivos/próximo passo com a última troca exatamente uma vez.
4. Rewriter lento/falho não acrescenta delay ao rollover; restore usa checkpoint+delta.
5. `burnless pilot --report` mostra host/versão/estratégia/contexto/confiança.
6. Kill -9, resize storm, Ctrl-C e exception restauram terminal e preservam journal/checkpoint.
7. Suite unitária/fake-host é obrigatória em CI; smoke real Claude+Codex é gate de release documentado e versionado.
8. `scripts/public_git_check.sh` e `.venv/bin/python -m pytest` passam antes de publicar.
9. Benchmark do relay reporta latência/CPU/bytes sem alterar o stream; journal/restore reportam p50/p95 e zero chamadas LLM no caminho crítico.
10. Cenário longo por host (20+ turnos, 3 rollovers) passa o golden harness P1 de `must_remember`/`must_forget` e mantém o payload dentro do budget.

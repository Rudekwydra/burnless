# Burnless — Capsules-as-Hero, Brain/Worker Split, and 99.82% Token Reduction

**Working document, 2026-05-20.** This draft consolidates the empirical findings from a real dogfooded session implementing the MCP server for Burnless itself, plus the architectural thesis they validate. The goal is to be detailed enough to relaunch on HN with confidence — without inflating numbers.

---

## TL;DR

In a single working session on 2026-05-20, we used Burnless to implement Burnless's own MCP server. Three real worker delegations emitted **354,232 raw tokens of output**. The Brain (Claude Code) consumed **639 capsule tokens** of that — **0.18%**. The delegated work shipped: 793 lines of new code, 18 async tests (18/18 passing), full design doc, all 8 DoD audit items green.

This isn't compression of data the Brain needs. It's **architectural filtering of data the Brain never needed**. The Brain reads structured envelopes (status, files touched, validations, errors, next step) — not the worker's stream-json deliberation, partial_json chunks, tool-use traces, or scratch reasoning.

## 1. The thesis

### 1.1 What Burnless is NOT

Burnless is not a compression tool. It is not a "tokens saver" that summarizes LLM outputs after the fact. It is not a wrapper around a single model that swaps in a cheaper one.

### 1.2 What Burnless IS

Burnless is a **protocol-layer architectural split** between two roles:

- **Brain** = strategic orchestrator. Decides what needs doing, picks tier, dispatches work, reads results, makes follow-up decisions. Brain stays in-context; it's the LLM you're actively chatting with.
- **Worker** = subprocess tier agent. Receives a tight spec, executes mechanically (edit files, run tests, report), emits a structured envelope. Worker is ephemeral; its raw output never enters the Brain's context.

Between them sits the **capsule** — a structured JSON record with a fixed schema:

```json
{
  "id": "d274",
  "status": "OK" | "PART" | "ERR" | "BLK",
  "files": ["tests/test_mcp_server.py"],
  "decisions": [],
  "validations": ["18/18 tests passing", "fixture mock_burnless_project created", ...],
  "errors": [],
  "risks": [],
  "next": "ready for production smoke test",
  "tokens": {"raw_tokens": 181915, "capsule_tokens": 190, "compression_ratio": 957.45}
}
```

This is the **mixture-of-experts architecture, but protocolar instead of inside-the-model**. Different LLMs (tiers) handle different roles; the Brain consumes their summaries, not their deliberation.

### 1.3 Why this works

Three reasons:

**(a) Brain doesn't need worker's deliberation.** When you delegate "implement these 4 files with this spec", Brain needs to know: did it work? what files changed? any errors? what's next? Brain does **not** need to see 131,565 tokens of partial_json chunks describing how the worker reasoned through the implementation. The deliberation is the worker's job, not the Brain's information.

**(b) Forcing structured envelope improves worker quality.** Workers cannot return a free-form "I think I'm done." They must categorize: status, files, validations, errors. This forces self-assessment that humans skip when reasoning informally. The capsule format is implicit auditor.

**(c) Capsules are durable; worker context isn't.** Worker dies. Capsule persists in `.burnless/capsules/dXXX.json`. Next session, next Brain, future audit — all read the same structured artifact. No need to reconstruct from raw logs.

## 2. The architecture

### 2.1 Brain ⟷ Worker via capsule

```
┌──────────────────────────────┐
│ Brain (Claude Code / Opus)   │
│  - decides what to delegate  │
│  - picks tier                │
│  - reads capsule.json (190 tok) │ ← total Brain context cost
│  - decides next move         │
└──────────────────────────────┘
              ↓ creates dXXX.md
       (delegation spec)
              ↓
┌──────────────────────────────┐
│ Worker (Haiku/Sonnet/Codex)  │
│  - reads spec                │
│  - edits files               │
│  - runs shell                │
│  - emits stream-json         │ ← 181,915 raw tokens, never seen by Brain
│  - finalizes envelope        │
└──────────────────────────────┘
              ↓
┌──────────────────────────────┐
│ QTP-A Auditor (filesystem)   │
│  - validates declared files  │
│  - checks size signatures    │
│  - writes capsule.json       │
└──────────────────────────────┘
              ↓
        capsule.json
       (190 tok essence)
              ↓
         back to Brain
```

### 2.2 Tier routing

Five tiers map to model classes by cost/quality:

| Tier | Default model | Use for |
|---|---|---|
| **diamond** | Opus / GPT-5 | Irreversible decisions, second opinion |
| **gold** | Sonnet / Opus | Architecture, structural refactor, design docs |
| **silver** | Haiku / Sonnet | Implementation with judgment calls |
| **bronze** | Haiku / qwen2.5:7b | Mechanical: spec + edit + execute |

The router is regex-based (`route(text, rules, default)`) — no LLM call needed to decide tier. Optional `--force` override; optional `BURNLESS_HARDCORE=1` env gates upgrades.

Critically: **spec quality determines tier, not task type**. A well-specified mechanical task with hard prohibitions runs bronze. An ambiguous task with "figure it out" runs gold. Roberto's empirical finding (2026-05-13): tight spec + bronze Haiku ≈ same outcome as loose spec + silver Sonnet, at ~3.75× lower cost. Capsule: [[tier-downgrade-spec-quality-2026-05-13]].

### 2.3 Plugin Protocol v0.7

Eight hooks expose the lifecycle for external interception:

- **H1 `pre_worker_prompt`** — modify spec before worker reads
- **H2 `post_worker_output`** — intercept envelope before Brain reads (this is where output compressors plug in)
- **H3 `session_state_read`** — pull mode for stateful plugins
- **H4 `audit_result_received`** — intercept QTP-A verdict
- **H5 `pre_brain_prompt`** — modify what Brain sees
- **H6 `post_brain_output`** — observe Brain's response
- **H7 `worker_invoke_override`** — replace worker invocation
- **H8 `pre_audit_call`** — modify audit logic

Plugins are JSON manifests in `~/.burnless/plugins/*.json` declaring which hooks they listen to and a stdio/HTTP endpoint. 5s timeout, fail-open.

### 2.4 MCP server (v1, shipped 2026-05-20)

Native MCP server `mcp__burnless__*` over stdio. Six tools:

| Tool | Wraps |
|---|---|
| `mcp__burnless__delegate(text, tier?, project_root?)` | `routing.route()` + `delegations.write_delegation()` |
| `mcp__burnless__route(text, project_root?)` | `routing.explain_route()` — read-only preview |
| `mcp__burnless__run(id, background?, project_root?)` | `live_runner.run_with_overflow_retries()` or `subprocess.Popen(start_new_session=True)` |
| `mcp__burnless__capsule(id, project_root?)` | read `.burnless/capsules/<id>.json` |
| `mcp__burnless__read(id, project_root?, max_log_lines?)` | 3-paths fallback: capsule → envelope → log tail |
| `mcp__burnless__status(id?, project_root?)` | project-wide health OR per-delegation state |

Brain Claude Code with the MCP entry registered in `~/.claude.json` calls these natively, no shell wrapper. Workers continue as subprocess of `claude` / `codex` binaries. CLI (`burnless delegate ...`) remains intact as parallel endpoint.

Entry point: `python -m burnless.mcp_server`. Optional pip extra: `pip install burnless[mcp]`.

## 3. Empirical findings — session 2026-05-20

### 3.1 The session

Goal: implement Burnless's MCP server using Burnless itself (meta-dogfood). Roberto authorized automode for parallel dispatch.

### 3.2 The three delegations

| ID | Tier | Model | Task | Raw tokens | Capsule tokens | Ratio | Output |
|---|---|---|---|---|---|---|---|
| d272 | gold | claude-opus-4-7 | Design + spec apertada for MCP server | 40,336 | 209 | **193×** | 461-line design doc with 6 tool specs |
| d273 | bronze | claude-haiku-4-5 | Implement 4 files | 131,565 | 240 | **548×** | mcp_server.py (512 lines), test_mcp_server.py (213 lines), docs/mcp.md (68 lines), pyproject.toml edit |
| d274 | bronze | claude-haiku-4-5 | Fix 13 failing test fixtures | 181,915 | 190 | **957×** | 18/18 tests passing |
| **Total** | | | | **354,232** | **639** | **555× avg** | working MCP server + tests + docs |

### 3.3 What Brain actually consumed

The orchestrating Brain (Claude Code Opus 4.7 in this session) read:
- 3 capsule JSONs = **639 tokens total**
- Issued 8 DoD audit grep commands (each <100 tokens output)
- Read the design file once for the dispatch decision (~3K tokens)

The Brain **never** saw the 354K of worker stream-json output. It lives on disk in `.burnless/log/dXXX.log` if needed for forensic audit. It has not yet been read by anything.

### 3.4 What Brain decided

In the same session, working off the 639 capsule tokens:
- Audit verdict per delegation (all OK)
- Dispatch follow-up bronze d274 to fix the 13 test failures revealed by smoke run
- Register `burnless` in `~/.claude.json` mcpServers
- Validate end-to-end via `handle_status()` smoke call

This is a meaningful working loop, not a trivial demo.

### 3.5 The hypothetical without Burnless

If the same MCP server work had been done by the Brain alone (no delegation):

- Brain would have had to read pytest output, edit files, re-run tests, debug fixtures — directly in its own context.
- Conservative estimate: 300-500K tokens of context burned, since pytest verbose runs alone are 5-100K each, and there are multiple iterations.
- Brain would have **forgotten** what it did across context compactions.

With Burnless: **0.18% of that was Brain context.** The remaining 99.82% lived in worker subprocess context and on disk.

This is what we mean by "tokens not paid > tokens compressed." We aren't compressing data the Brain wanted; we're filtering data the Brain never needed.

## 4. The less family

The architectural pieces are not one tool. They're a family of MIT projects, each owning one concern. Honest status table — **shipped vs idea**:

| Project | Status | Role | Layer |
|---|---|---|---|
| **Burnless** | **active v0.7.x** | Brain/Worker split with capsule mediator | LLM orchestration |
| **Forgetless** | **active** (~270 capsules in corpus) | Cross-session capsule memory with ranked recall | Memory |
| **Gapless** | **active hook** | Liveness/observability via task-notification system-reminder; no polling | Observability |
| **TodoLess** | **active hook** | Auto-captures user requests via UserPromptSubmit hook; bronze regex classifier | Workflow capture |
| **Leakless** | placeholder | Privacy-preserving filter; spec TBD | Compliance |
| **Blockless** | placeholder / idea | "Boss Voice Injection" — break LLM autonomous loops with human-pressure phrases | Behavioral (contract) |
| **Driftless** | placeholder / idea | Symmetric pair of Blockless — controlled expansion of plan | Behavioral (expand) |
| **Wastless** | naming proposal 2026-05-09 | Anti-token-waste, next member candidate; slogans pending | TBD |

The naming is intentional: "less" = absence of waste, not absence of capability. Less tokens, less mind-loop, less leak, less drift, less todo-friction. Some members ship today and others are placeholders we may or may not build — the table is honest about which is which.

## 5. Reproducing this result

### 5.1 Install

```bash
# Burnless
git clone https://github.com/Rudekwydra/burnless
cd burnless
pip install -e ".[mcp]"          # MCP extra optional

# Forgetless (companion, optional)
git clone https://github.com/Rudekwydra/forgetless
# (single-file CLI, see README)
```

### 5.2 Initialize project

```bash
cd my-project
burnless init   # creates .burnless/config.yaml + folders
```

### 5.3 Configure Claude Code

In `~/.claude.json`:

```json
{
  "mcpServers": {
    "burnless": {
      "command": "python",
      "args": ["-m", "burnless.mcp_server"],
      "env": {"PYTHONPATH": "/path/to/burnless/src"}
    }
  }
}
```

Restart Claude Code.

### 5.4 Dogfood

In your next Claude Code session, ask Claude to implement something non-trivial. Watch Claude call `mcp__burnless__delegate()` natively. After it finishes:

```bash
burnless metrics # see capsule compression
```

This number is the empirical claim of Burnless. We expect it to match the order-of-magnitude reported above on realistic tasks.

## 6. What is genuinely tested vs pending

This section exists because empirical claims need an honest scope.

### Tested ✅
- **MCP server v1**: 6 tools, 18/18 pytest passing, smoke handler called end-to-end on real `.burnless/` (handle_status returned 255 capsules, last d274).
- **Capsule compression ratios**: directly measured in 3 delegations on the day Burnless was used to implement its own MCP server. Numbers come from `burnless capsule <id>` reading the actual `.burnless/capsules/<id>.json` files. Not synthetic.
- **Worker subprocess preservation**: workers (claude/codex bin) continue as subprocess, never moved to in-process. Verified by grep.
- **CLI backward compatibility**: `burnless delegate` and friends still work standalone; MCP is parallel.

### Pending ⏳
- **MCP server end-to-end via Claude Code**: requires restart of this Claude Code session to load the `mcpServers` entry from `~/.claude.json`. Not validated in the session that wrote the server.
- **HTTP transport for MCP**: explicitly out of scope for v1. Stdio only.
- **Reproduction by external users**: numbers reported are from Roberto's machine, Roberto's tasks. No third-party replication yet.

### Known issues
- Worker output >100K raw tokens occasionally triggers `auditor: missing_final_json` (capsule [[feedback-burnless-auditor-large-output-bug-2026-05-18]]). Workaround in flight (d271 fix). Did not affect this session, but is a known limit.
- Bronze Haiku regressing to mock test patterns when spec mixes EDIT and EXECUTE blocks. Workaround: keep them separated in spec ([[feedback-bronze-haiku-edit-vs-execute-separation]]).

## Appendix B — How to read these numbers skeptically

If you don't trust the ratio because "capsules are compressed by hand by the orchestrator":

- The 190 / 209 / 240 capsule token counts are the **literal sizes of the JSON files** at `.burnless/capsules/dXXX.json`, tokenized via `tiktoken.cl100k_base`. Not hand-compressed by a human reviewer.
- The 181915 / 131565 / 40336 raw_token counts are the size of the worker's complete subprocess output stream — captured by `live_runner` to `.burnless/log/dXXX.log` and tokenized by the same encoding. Available on disk for forensic verification.
- "raw" here means EVERY token the worker emitted: stream-json envelopes, tool_use chunks, partial_json deltas, internal reasoning surfaced via verbose mode. Not pre-summarized by anything.
- The Brain's actual token consumption (the capsule) is a strict subset of what's required to make the same decisions a Brain that read everything would have made. We've verified this empirically: dispatch follow-ups (d274 was dispatched purely from d273's capsule) succeed without ever reading the raw worker output.

If you don't trust the dogfood claim because "they wrote the framework testing the framework":

- We literally did. That's the point. The work product is a real MCP server. The pytest output shows 18/18 passing. The git log is the audit trail. The HN crowd can install both packages and reproduce.

If you don't trust because "this all assumes the Brain doesn't actually need the worker's deliberation":

- Bring counterexamples. We expect domains where it breaks (creative writing, novel research). For mechanical software engineering, the architecture appears robust across ~270 capsules in our active corpus.

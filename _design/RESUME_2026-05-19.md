## Product Thesis

Burnless and Synapsis are related, but not the same product.

- Burnless makes LLM work cheaper, auditable, and routable across tiers.
- Synapsis makes LLM work continuous across entities, projects, and models.
- Forgetless-style search/ranking over accumulated capsules is a Pro/Cloud
  intelligence layer, not a core MIT requirement.

The open source strategy is open-core:

1. MIT CLI earns trust through a complete protocol, reproducible benchmarks,
   plugins, local operation, and honest claims.
2. Desktop Pro earns money through convenience, multi-LLM continuity,
   better compression, clean UX, and quota peace.
3. Team/Cloud earns money through memory operations: Capsule SEO, ranking,
   retention, dashboards, policies, audit logs, SSO/RBAC, and SLA.

Do not try to monetize the protocol itself too early. The protocol being open
is the credibility moat. Charge for operation, distribution, UX, governance,
and managed intelligence.

## MIT Package Boundary

The MIT package should be strong enough that developers, reviewers, and
influencers trust the project.

MIT includes:

- `burnless` CLI: Brain, Workers, tiers, routing, capsules, audit, plugin
  protocol, metrics.
- Full local reference compression plugin:
  `examples/plugins/burnless-compress`.
- Local capsule creation, storage, reading, and decoding.
- Reproducible benchmarks and math: `bench/run.py`, `bench/v2.py`, `MATH.md`.
- Provider-neutral Worker model: Claude, Codex, Gemini, OpenAI, Ollama, etc.
- Basic metrics, including global aggregation once d272 lands.
- Public docs that avoid overclaiming and frame capsules as semantic capsules,
  compressed state representations, or dense semantic summaries.

MIT should not include the official managed implementation of:

- Capsule SEO/search/ranking over accumulated history.
- Salience-weighted retention.
- Server-side metadescription pipeline.
- Team dashboards and cross-project knowledge graph.
- Managed workload tuning.
- Drift monitoring.
- Multi-tenant glossary management.
- KMS/HSM, SSO/RBAC, legal hold, audit log retention, SLA.
- Polished proprietary Desktop app.

Third parties may implement these through the plugin protocol. The official
managed version is Pro/Cloud.

## Pro Package Boundary

Pro is not "Burnless with a better compressor". Pro is the packaged product:

> Use Claude, GPT, Gemini, and local models with continuous context, less quota
> waste, and visible proof of savings.

Suggested levels:

| Tier | Audience | Product Promise |
|---|---|---|
| MIT CLI | Developers, researchers, hackers | Open protocol, self-host, reproducible proof |
| Desktop Free | Non-developer users | One-click app, basic savings, visible meters |
| Desktop Pro | Power users | Better compression, multi-LLM continuity, calm UX |
| Pro Privacy | Privacy-sensitive users | Local-only capsules/index, encrypted local history |
| Team Cloud | Small teams | Shared memory, dashboards, policies, auditability |
| Enterprise | Companies | SSO/RBAC, retention, legal hold, KMS/HSM, SLA |

Prefer simple subscription pricing first. Rev-share on measured savings is
strategically elegant but operationally harder to explain, audit, and sell.
Keep rev-share as a later heavy-user/team option.

## Empirical Worker Pattern

Roberto's local empirical finding: the audit loop was not the best primary
quality mechanism. It often looped because workers completed the work but did
not document evidence in the exact shape the auditor expected.

Current operating pattern:

- Gold/Brain writes detailed delegation specs.
- Specs include hard prohibitions and explicit Definition of Done.
- Silver and Bronze workers can both run Haiku for daily local work.
- Haiku workers delivered well when the spec was tight.
- Sonnet is currently retired from Roberto's daily local workflow.
- Audit should be defensive and lightweight, not the centerpiece of quality.

Product implication: sell Burnless Free as spec-driven delegation, tier routing,
capsules, metrics, and optional evidence checks. Do not over-position audit as
the main thing that makes work reliable.

## Capsule SEO / Metadescription Design

Roberto's proposed Pro idea is valid:

1. Client sends a capsule/event to Pro Cloud.
2. Cloud generates a metadescription, labels, salience score, graph features,
   dedup key, retention decision, and searchable index entries.
3. Cloud stores the managed metadata and returns an opaque unique ID.
4. Client stores the local capsule plus `cloud_ref`.

The server should not return the internal SEO method. It returns only the
minimum UX payload, for example:

```json
{
  "capsule_id": "cap_01J...",
  "meta_id": "meta_01J...",
  "salience": 0.84,
  "retention": "keep",
  "labels": ["decision", "backend", "auth"],
  "public_hint": "Backend auth decision"
}
```

Protect the moat by keeping these server-side:

- metadescription prompts
- embeddings or feature vectors
- scoring formulas
- graph edges used for ranking
- retention policy internals
- model ensemble/tuning logic

Use opaque IDs such as ULID/UUIDv7 for public references. Use tenant-scoped
HMACs for deduplication so identical content across tenants does not produce a
shared public fingerprint.

## Privacy Modes

Privacy is a product axis, not a universal promise.

| Mode | Cloud Receives | Efficiency | Promise |
|---|---|---|---|
| Pro Standard | capsule or full semantic summary | highest | maximum savings and managed memory |
| Team Cloud | capsule + team/project metadata | highest | shared memory and governance |
| Pro Privacy | no raw capsule; local index only or redacted optional payload | lower | maximum local sovereignty |
| Pro Privacy+ | local encrypted advanced index | medium-high | privacy with better local retrieval |

Do not claim that Pro Privacy gives both maximum privacy and maximum savings.
The honest product copy is:

```text
Savings mode uses Burnless Cloud to index capsules and retrieve only the most
relevant context.

Privacy mode keeps capsules and the memory index on this device. It uses more
tokens and sends less data.
```

This is not a weakness. It is a clear choice: maximum savings or maximum
privacy.

Synapsis currently exists as a small Tier 1 prototype in
`~/antigravity/synapsis`.

Current Synapsis behavior:

- Detects entity/context switch through Active Capsule Routing.
- Surfaces relevant context through Forgetless rank.
- Captures turns into the active entity store.
- Produces an enriched prompt for Claude/GPT/Ollama/etc.

Synapsis is the larger product vision: one linear workspace chat with active
memory management across all entities, topics, and models. It should not be
launched as the main product yet. It is too prototype-shaped.

Use Synapsis ideas inside Burnless Pro:

- Active Capsule Routing.
- Capsule SEO.
- Memory graph.
- Cross-model continuity.
- Salience/retention.

Do not mix Synapsis into the public Burnless README as a product claim yet.
The clean public line is:

```text
Burnless makes LLM work cheaper and auditable.
Synapsis makes LLM work continuous.
```

## Free Savings Formula

Free savings must be shown as a decomposition, not a single magic number.

Mechanisms:

- Input compression: raw chat text becomes compact text before the LLM.
- Maestro history/cache: Maestro receives capsules/compact state instead of a
  growing raw transcript, and stable prefix cache hits are counted when the
  provider reports them.
- Worker one-shot isolation: workers receive one clean spec/task, not the full
  prior chat context. Raw worker stdout/stderr stays in logs and is not replayed
  into Maestro.
- Tier routing: tight Gold/Brain specs can route execution to cheaper Haiku
  workers instead of using gold-tier reasoning for every step.

The user-facing metrics should show these buckets:

```text
Free savings breakdown:
  input compression      N   (txt -> compact prompt)
  Maestro history/cache  N   (linear capsules + warm prefix)
  worker one-shot        N   (worker logs not replayed)
  tier routing           N   (cheap worker vs gold)
  total                  N
```

This is the core Free story: the economy is moderate on the current message,
large on Maestro's accumulated history, and often very large on workers because
they do not carry previous chat context at all.
